# vendor.xmldoc — генерация XML УПД из Битрикс24

D7-модуль для коробочного Битрикс24.

## Быстрый старт (v1.4.0)

1. Установите модуль → **Обновить** до 1.4.0
2. Настройки: `/bitrix/admin/settings.php?mid=vendor.xmldoc` (entityTypeId СП = 31 по умолчанию)
3. Заполните **Мои реквизиты** и реквизиты компании-покупателя
4. Откройте сделку → **Сформировать УПД**
5. Робот CRM: действие **Сформировать УПД (XML)**
6. История: **Настройки → XML УПД → История документов**

## Установка

1. Скопируйте папку `local/modules/vendor.xmldoc` в корень сайта Битрикс.
2. Откройте в админке: **Marketplace → Установленные решения**  
   или прямой URL: `/bitrix/admin/partner_modules.php?lang=ru`
3. Найдите модуль **«Генерация XML (УПД)»** в списке локальных решений.
4. Нажмите **Установить**.

## Настройки модуля

После установки модуль появится в:

**Настройки → Настройки продукта → Настройки модулей → vendor.xmldoc**

Прямой URL: `/bitrix/admin/settings.php?mid=vendor.xmldoc&lang=ru`

Заполните:

- ID реквизита продавца
- ID подписанта
- entityTypeId СП «Счета»
- ключ DaData (опционально)
- путь к XSD (опционально)

## Если модуль не виден в списке

1. Проверьте путь: `/{корень сайта}/local/modules/vendor.xmldoc/install/index.php`
2. Проверьте права на папку — веб-сервер должен читать файлы.
3. Убедитесь, что установлен модуль **CRM**.
4. Очистите кеш: **Настройки → Производительность → Очистить кеш**.

## Использование

В карточке сделки или СП «Счета» — кнопка **«Сформировать УПД»**.

## Бизнес-процесс и робот CRM

### Activity в дизайнере (рекомендуется)

После обновления модуля до **1.3.0** в роботах CRM и БП доступно действие **«Сформировать УПД (XML)»** (категория «Генерация XML (УПД)»).

**Где:** CRM → Сделки → Роботы → Добавить → Действия → **Сформировать УПД (XML)**.

Activity возвращает: `Success`, `FileId`, `FileName`, `Version`, `Message`, `Errors`.

### PHP-код (запасной способ)

```php
\Bitrix\Main\Loader::includeModule('vendor.xmldoc');
return \Vendor\Xmldoc\Bp\Runner::fromDeal({=Document:ID});
```

## Карта полей

| XML | Источник B24 |
|-----|--------------|
| НомерДок | ID сделки / номер счёта СП / UF_UPD_NUMBER |
| ДатаДок | Дата генерации |
| СвПрод | Мои реквизиты |
| СвПокуп | Реквизиты компании + DaData |
| Товары | ProductRowTable |
| Подписант | Настройки или текущий пользователь |

Образцы: `edo_6608_*.xml` в корне модуля.

## Версионирование и кодировка

- **UF_UPD_FILE:** в карточке всегда **актуальная** версия.
- **Файлы:** каждая генерация — новый `CFile` (`УПД_{ID}_v{N}.xml`); **старые версии не удаляются**.
- **История:** метаданные в `b_xmldoc_document` (VERSION, FILE_HASH, DOC_STATUS, FILE_ID).
- **Статусы:** `generated` при создании; `sent` / `accepted` / `rejected` — этап 2 (ЭДО).
- **Кодировка:** XML в UTF-8 → файл в **windows-1251** (настраивается в options.php).

Подробный аудит: [AUDIT.md](AUDIT.md)

## Требования

- PHP 8.1+
- Битрикс 17+ (D7)
- Модуль CRM
- Расширение PHP `iconv` (конвертация UTF-8 → windows-1251)

### Шаг 1. Настройки модуля — что вводить

Откройте **Настройки модулей → vendor.xmldoc** и заполните:

| Поле | Что ввести | Для первого теста |
|------|------------|-------------------|
| **Ключ DaData API** | Токен с [dadata.ru](https://dadata.ru) | Можно оставить пустым, если реквизиты компании заполнены полностью |
| **ID реквизита продавца** | Числовой ID из `b_crm_requisite` | **Оставить пустым** — подтянутся «Мои реквизиты» портала |
| **Подписант** | `Из настроек` или `Текущий пользователь` | Для теста: `Текущий пользователь` |
| **ID подписанта** | ID пользователя Битрикс (если режим «Из настроек») | Например, `1` — администратор |
| **Должность подписанта** | Текст для XML | `Сотрудник` или реальная должность |
| **entityTypeId СП «Счета»** | ID типа смарт-процесса | `31` (уточните у админа CRM, если счета в другом СП) |
| **Функция документа** | Код функции УПД | `СЧФДОП` (как в образце Diadoc) |
| **Путь к XSD** | Абсолютный путь на сервере | Пока **пусто** — без XSD-валидации |
| **Публиковать в таймлайн** | Галочка | Включить — удобно видеть результат в карточке |

Нажмите **Сохранить**.

### Шаг 2. Продавец — «Мои реквизиты»

**Где:** CRM → **Настройки** (шестерёнка) → **Мои реквизиты**  
или **CRM → Настройки → Реквизиты → Моя компания**.

Заполните реквизит, который будет продавцом в УПД:

| Поле в B24 | Обязательно для теста | Пример (как в образце `edo_6608`) |
|------------|----------------------|-----------------------------------|
| Тип | ИП или ЮЛ | ИП |
| Фамилия, Имя, Отчество | Для ИП — да | Кондаков Андрей Владимирович |
| ИНН | **Да** | `246522779824` |
| КПП | Для ЮЛ (ИНН 10 знаков) | — |
| Юридический адрес | **Да** | Индекс, регион, город, улица, дом, квартира |

Адрес лучше заполнять **структурированно** (индекс, регион, город, улица, дом) — модуль собирает узел `АдрРФ` из полей адреса реквизита.

**Проверка:** если «ID реквизита продавца» пустой, модуль ищет компанию «Мои реквизиты» (`EntityLink::getDefaultMyCompanyId` или `IS_MY_COMPANY=Y`) и берёт первый реквизит этой компании.

### Шаг 3. Покупатель — компания в CRM

**Где:** CRM → **Компании** → создать или открыть компанию покупателя.

#### 3.1. Карточка компании

| Поле | Что ввести |
|------|------------|
| Название | `ООО "СОНЯ-ТЕКСТИЛЬ"` (или тестовое юрлицо) |
| Привязка к сделке | В сделке в поле **Компания** выбрать эту компанию |

#### 3.2. Реквизиты компании

**Где:** вкладка **Реквизиты** в карточке компании → добавить реквизит.

| Поле | Обязательно | Пример |
|------|-------------|--------|
| Полное наименование | **Да** | `ООО "СОНЯ-ТЕКСТИЛЬ"` |
| ИНН | **Да** | `2404008231` |
| КПП | **Да для ЮЛ** (ИНН 10 цифр) | `246201001` |
| Юридический адрес | **Да** | Красноярск, проспект..., дом 30А |

Без адреса генерация остановится с ошибкой: *«Юридический адрес покупателя»*.

#### 3.3. DaData (опционально)

Если ИНН есть, а адрес/КПП пустые — укажите ключ DaData в настройках модуля. При генерации модуль запросит данные и **дозапишет** их в реквизит компании.

**Тест без DaData:** заполните ИНН, КПП и адрес вручную в реквизите.

### Шаг 4. Тест через сделку (основной сценарий)

Рекомендуемый тест — существующая сделка из воронки **Сервис** (`CATEGORY_ID = 0`) или **Розница** (`CATEGORY_ID = 1`), например сделка **6608** из образца.

#### 4.1. Подготовка сделки

**Где:** CRM → **Сделки** → открыть сделку.

| Поле сделки | Что указать |
|-------------|-------------|
| **Компания** | Компания-покупатель из шага 3 |
| **Товары** | Минимум 1 позиция (см. ниже) |
| **Номер УПД (1С)** `UF_UPD_NUMBER` | Пусто → в XML попадёт **ID сделки**; или введите `УПД-6608` для проверки приоритета UF |
| Воронка | Любая из рабочих (0 или 1) |

#### 4.2. Товарные позиции

**Где:** блок **Товары** в карточке сделки → **Добавить товар**.

Пример строки (как в реальной отгрузке):

| Поле | Значение |
|------|----------|
| Наименование | `Ремонт стиральной машины` |
| Количество | `2` |
| Ед. изм. | `н/ч` или `час` (код ОКЕИ `356`) |
| Цена | `2500` |
| НДС | `22%` (или ставка вашего портала) |

Без товаров ошибка: *«Товарные позиции»*.

#### 4.3. Генерация

1. Откройте карточку: `/crm/deal/details/{ID}/`
2. В правой части тулбара нажмите **«Сформировать УПД»**
3. Подтвердите в диалоге
4. Должно появиться уведомление и открыться скачанный XML

Если кнопки нет:

1. **Обновите модуль** до 1.1.1: `/bitrix/admin/partner_modules.php` → **Генерация XML (УПД)** → **Обновить** (или переустановить).
2. Очистите кеш: **Настройки → Производительность → Очистить кеш**.
3. Кнопка появляется **только в тулбаре** справа вверху карточки.
4. В консоли браузера (F12) проверьте: `window.XMLDOC_CONFIG` — должен содержать `entityId` и `ajaxUrl`.
5. Запрос идёт на `/local/modules/vendor.xmldoc/ajax/generate.php` (основной способ).
6. Запасной action (если нужен): `vendor.xmldoc:upd.generate`.

#### 4.4. Что проверить после успеха

| Проверка | Где смотреть | Ожидаемый результат |
|----------|--------------|---------------------|
| Файл в сделке | Поле **Файл УПД** | `УПД_{ID}.xml` |
| Имя файла | Скачанный файл | `УПД_6608.xml` для сделки 6608 |
| Номер в XML | Узел `СвСчФакт/@НомерДок` | ID сделки или значение `UF_UPD_NUMBER` |
| Дата в XML | `СвСчФакт/@ДатаДок` | **Сегодняшняя дата** (дата генерации) |
| Продавец | `СвПрод` | ИНН и ФИО из «Мои реквизиты» |
| Покупатель | `СвПокуп` | ИНН, КПП, название компании |
| Кодировка | Первая строка файла | `encoding="windows-1251"` |
| Таймлайн | Лента сделки | Комментарий «Сформирован документ … (версия 1)» |
| История в БД | Таблица `b_xmldoc_document` | Строка с `VERSION=1`, `ENCODING=windows-1251` |
| Лог | Таблица `b_xmldoc_log` | `STATUS=success` |

#### 4.5. Сравнение с образцом

В корне модуля лежит эталон: `edo_6608_2026-06-15_07-29 (2).xml`.

Сверьте для той же сделки 6608:
- структуру `СвИП` / `СвЮЛУч`
- блоки `ГрузОт`, `ГрузПолуч`, `СвПродПер`
- суммы и НДС по строкам товаров
- `ВерсФорм="5.03"`, `Функция="СЧФДОП"`

Полное побайтовое совпадение не обязательно: дата, время (`ВремИнфПр`) и `ИдФайл` будут другими.

#### 4.6. Тест версионирования

1. Не меняя сделку, нажмите **«Сформировать УПД»** ещё раз
2. В `UF_UPD_FILE` — **тот же** файл по имени, но новый `file_id`
3. В `b_xmldoc_document` — **две** строки: `VERSION=1` и `VERSION=2`
4. В таймлайне — комментарий с «версия 2»

#### 5.1. Подготовка полей СП

Откройте: `/crm/type/31/details/{ID}/` → кнопка **«Сформировать УПД»**.

### Шаг 7. Негативные тесты (проверка ошибок)

| Сценарий | Как воспроизвести | Ожидаемое сообщение |
|----------|-------------------|---------------------|
| Нет компании | Убрать компанию из сделки | ИНН / наименование / адрес покупателя |
| Нет реквизитов компании | Компания без реквизита | «ИНН покупателя», «Наименование покупателя» |
| ЮЛ без КПП | ИНН 10 цифр, КПП пустой | «КПП покупателя» |
| Нет товаров | Пустой блок товаров | «Товарные позиции» |
| Нет «Мои реквизиты» | Не заполнены реквизиты портала | «ИНН продавца», «Наименование продавца» |
| Нет подписанта | Режим «Из настроек», ID=0, у пользователя пустое ФИО | «ФИО подписанта» |

Текст ошибки в интерфейсе:  
*«Не удалось сформировать УПД. Отсутствуют обязательные данные: …»*

Детали также пишутся в `b_xmldoc_log` (`STATUS=validate` или `error`).

### Шаг 8. Проверка кодировки

1. Откройте скачанный XML в редакторе с выбором кодировки (Notepad++, VS Code).
2. Укажите **windows-1251** — кириллица должна читаться нормально.
3. При открытии как UTF-8 текст будет «кракозябрами» — это нормально для файла под Диадок.

Проверка в БД:

```sql
SELECT ENTITY_TYPE, ENTITY_ID, VERSION, ENCODING, FILE_HASH, CREATED_AT
FROM b_xmldoc_document
WHERE ENTITY_TYPE = 'deal' AND ENTITY_ID = 6608
ORDER BY VERSION;
```

### Шаг 9. Прямой вызов AJAX (для отладки)

В консоли браузера на странице сделки (подставьте `entityId` и `sessid` из `window.XMLDOC_CONFIG`):

```javascript
BX.ajax({
    url: '/local/modules/vendor.xmldoc/ajax/generate.php',
    method: 'POST',
    dataType: 'json',
    data: {
        sessid: window.XMLDOC_CONFIG.sessid,
        entityType: 'deal',
        entityId: 6608
    },
    onsuccess: console.log
});
```

Успешный ответ:

```json
{
  "success": true,
  "fileId": 12345,
  "fileName": "УПД_6608.xml",
  "fileUrl": "/upload/xmldoc/...",
  "version": 1,
  "encoding": "windows-1251"
}
```
